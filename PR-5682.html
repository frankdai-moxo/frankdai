API investigation — can each provider do what we do today?

"What we do today" in TIME_BOOKING = read host availability → render Moxo's own slot grid → book the meeting programmatically → expose start/end/join-URL/invitee as DDR. The question is which providers' APIs actually support that.

The clean cut: Calendly could be API-booked in-portal, but it shouldn't be — its branding, reschedule/cancel, custom questions, reminders and timezone handling are the reason a customer chose it. Treating both schedulers as hosted handoff makes the model one discriminator, not two axes: a provider is either a generic calendar (Moxo renders + books) or a dedicated scheduler (mint single-use link → hosted page → webhook, correlated by utm_campaign). Both schedulers reuse today's ZOOM_SCHEDULER executor verbatim.

The abstraction: one discriminator — providerKind

kind: 'calendar' — Google, Outlook

  • Raw free/busy data — no UI of its own.
  • Builder picks meeting type + duration.
  • Moxo renders the slot grid in-portal.
  • Moxo creates the calendar event → sync completion.

kind: 'scheduler' — Calendly, Zoom Scheduler

  • A product whose hosted booking UI is its value.
  • Host = the connection holder — no host role. Builder picks connection + schedule (type + length inherited).
  • Assignee books on the provider's hosted page, against the host's availability.
  • Mint single-use link → webhook completion (utm_campaign).

Because availability, booking UI, and completion path all follow from what the provider is, they collapse to a single registry discriminator providerKind: 'calendar' | 'scheduler' — resolved in one kind-complete switch + assertNever (CLAUDE.md unified-path rule). No two-axis matrix, no parallel step types. The builder is unified; the one intentional divergence is the assignee's booking screen.

The conditional-field rule: for calendar providers the builder picks a host role + meeting type + length + participants. For scheduler providers (Calendly and Zoom Scheduler share this exact builder shape) all of those collapse: the connection holder is the host (no role picker), and meeting type / length / invitees / title + agenda are owned by the selected schedule + the provider's hosted page. The scheduler config is essentially just connection + schedule (+ an optional DDR purpose blurb passed via UTM/prefill). The Calendly card doubles as the Zoom card — swap "Calendly event type" for "Zoom schedule". DDR output tokens are identical across all four providers so downstream steps never branch on who booked.

One switch (providerKind) ending in assertNever — adding a provider can't silently miss a branch. The scheduler path (blue) is today's ZOOM_SCHEDULER executor generalized to two providers: each supplies a createSingleUseLink() adapter + a webhook handler; the link-mint, UTM stamping, dispatch, and CAS completion are shared. Calendly differs from Zoom only in the two adapter calls.

What this buys us

  • ✅ One discriminator (calendar vs scheduler) — no two-axis matrix to reason about.
  • ✅ One step type, one builder, one DDR output shape (meet.*).
  • ✅ Calendly + Zoom ship as providers; the scheduler runtime is shared (link-mint + webhook), each adds two small adapters.
  • ✅ Retires the ZOOM_SCHEDULER step type while reusing its executor as the shared scheduler path.
  • ✅ We don't reimplement Calendly's booking UX (branding, reschedule, custom questions) — we defer to it. Less to build, truer to the product.

Open questions / risks

  • ⚠️ Schedulers leave Moxo — assignee books in a new tab, completion is webhook-driven (no polling). Acceptable + intended, but the local-dev story needs a webhook tunnel (already true for Zoom).
  • ⚠️ Calendly single-use link needs a paid Calendly plan + OAuth app — gate the provider, surface a clear message.
  • ⚠️ Calendly UTM-stripping edge case reported in some link formats — verify utm_campaign survives on the minted booking_url before relying on it for correlation (fallback: invitee URI lookup).
  • ⚠️ DDR output rename: today's ZOOM_SCHEDULER emits flat bookingTime/meetingLink/inviteeEmail; normalize to meet.*. Cheap now (unused), a migration later.
  • ✅ Host binding settled: for schedulers the connection holder is the host (no host role, no per-flow resolution) — matches today's ZOOM_SCHEDULER. Just map its connectionId + scheduleId into the unified booking.* shape.
  • ⚠️ Consequence of connection=host: the per-flow AWAITING_HOST_SETUP gate is calendar-only. A scheduler with an unconnected/expired connection is a build-time / config error (today's RECONNECT_REQUIRED → step FAILED), not a per-flow host gate.
  • ⚠️ Supersedes the existing ADR ("why a separate step type") — write the reversal.
