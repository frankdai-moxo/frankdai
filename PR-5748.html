1 · Why this PR exists

Modern hosted MCP servers don't ship pre-registered OAuth apps

Until this PR, Moxo's oauth-mcp profile assumed every MCP server worked like HubSpot: a developer registers a Moxo OAuth app in the provider's dev portal, we bake the CLIENT_ID + CLIENT_SECRET into AWS Secrets Manager, every customer's OAuth handshake uses that shared identity.

A new generation of MCP servers — Fathom, Linear's hosted MCP, Notion's hosted MCP, Cloudflare's MCP Workers template — does not work that way. They expose Dynamic Client Registration (RFC 7591) + PKCE-public-client: the client (us) registers itself with the provider's auth server at the first connect, there is no client_secret, and PKCE alone proves identity. Without DCR support, Moxo can't connect to any of these — they all 401 at the token endpoint.

Pre-registered confidential
What HubSpot needs
  • ● Human dev registers Moxo app in HubSpot's portal
  • ● CLIENT_ID + CLIENT_SECRET baked into Secrets Manager
  • ● Endpoints hard-coded in registry entry
  • ● Token exchange sends client_secret

Works today. Stays working — unchanged by this PR.

DCR + public client
What Fathom (and the next 4 hosted MCPs) need
  • ● Moxo registers ITSELF on first connect (per-org)
  • ● No secret ever issued — PKCE proves identity
  • ● Endpoints discovered from .well-known/oauth-authorization-server
  • ● Token exchange sends code_verifier only

What this PR adds. Same code path; selected per integration via two flags on providerOverrides.

2 · The DCR handshake, step by step

PKCE replaces client_secret — the verifier never travels on the wire until token exchange

DCR (RFC 7591) lets a client register itself with the authorization server at runtime, instead of a human pre-registering in a developer portal. PKCE (RFC 7636) gives that client a way to prove identity at the token endpoint without a secret — the client commits to a random code_verifier via its SHA-256 hash at the start of the flow, then reveals the original on token exchange. Anyone who intercepted the auth code can't redeem it without the verifier.

Phase A · Once per org
Discovery + DCR. Cached in mcp_oauth_dynamic_clients. Subsequent users in the same org reuse the row — one extra HTTP per org, ever.
Phase B · Per user
Standard PKCE auth-code flow. code_verifier stays in Moxo's signed state token; only its SHA-256 hash goes over the wire.
Phase C · Token exchange
No client_secret field anywhere. The verifier hash from Phase B is what the AS compares against — that's what proves we're the same client that started the flow.
3 · What the PR actually changes

One resolver, two flags, three new files — HubSpot path is byte-identical

Three new files

  • services/mcp-oauth-discovery.ts — fetches .well-known/oauth-authorization-server, validates each endpoint against the SSRF guard AND a per-integration allowlist. 60s in-memory cache.
  • services/mcp-dynamic-client.ts — ensureDynamicClient(orgId, integration, redirectUri) with a per-(org, integration) advisory lock so concurrent first-connects can't double-register.
  • db/schema.ts — new table mcp_oauth_dynamic_clients, composite PK on (orgId, integrationId), FK to organizations, classified identity in the ADR-188 firewall.

One surgical helper in oauth.ts

resolveOAuthClientMaterial(integration, orgId, options) replaces three duplicate getAppCredentials reads at:

  • generateIntegrationAuthUrl (kickoff)
  • handleIntegrationOAuthCallback (token exchange)
  • performRefreshTokenExchange (refresh)

Each call site stays one line. The DCR vs. legacy fork lives in one helper — per ADR-204 "one kind-complete resolver, not three per-consumer matchers."

The two new flags on providerOverrides

auth: {
  type: 'OAUTH2',
  oauth2: {
    authorizationUrl: 'https://fathom.video/mcp/oauth/authorize',   // kept as documentation
    tokenUrl:         'https://api.fathom.ai/mcp/oauth/token',      // runtime uses discovered values
    scopes: ['mcp'],
    providerOverrides: {
      requiresPkce: true,                                            // existing
      discoverOAuthEndpoints: true,                                  // NEW — read .well-known at runtime
      discoveryAllowedOrigins: [                                     // NEW — explicit allowlist (no same-eTLD)
        'https://api.fathom.ai',
        'https://fathom.video',
      ],
      useDynamicClientRegistration: true,                            // NEW — self-register on first connect
      publicClient: true,                                            // NEW — omit client_secret at /token
    },
  },
}

HubSpot's entry doesn't set any of the new flags → its codepath is byte-identical to today. Verified by 15 pre-existing OAuth integration tests still passing.

5 · Why this PR unlocks a class of MCP providers, not just one

OAuth 2.1 + DCR + PKCE-public-client is the de-facto standard for hosted MCP servers

The MCP spec recommends this exact OAuth profile for remote MCP servers, and the major hosted providers have aligned on it. Every connector below ships their own AS metadata + DCR endpoint + public-client profile. After this PR, adding any of them is a single registry entry — same shape as Fathom, different URLs.

Strategic shape

This is the same generalization arc as the unified integrations registry itself: the first connector is the expensive one because it forces the right abstraction; every subsequent connector lands as a registry edit. Going forward, "add MCP support for <provider>" is a half-day registry PR, not a multi-day plumbing project.

Security posture

Three defenses on the discovery + registration path

  1. SSRF guard on the metadata URL AND on every endpoint the AS returns (incl. the registration endpoint).
  2. Per-integration discoveryAllowedOrigins — an explicit list of scheme+host pairs. Same-eTLD is not accepted (a compromised subdomain on the same registrable domain would otherwise pass).
  3. S256 PKCE required — refuse to use an AS whose metadata doesn't advertise S256 in code_challenge_methods_supported.

DCR row is per-(org, integration) — deleting an org cascades to its DCR identity at the provider. Outbound fetch() uses redirect: 'manual' + rejects 3xx so a redirect can't bypass the SSRF/allowlist gate.

Out of scope (deliberate)

What this PR doesn't touch

  • ● No frontend changes — Fathom appears via the existing registry-driven Connected Apps UI.
  • ● No changes to HubSpot's or any other existing integration's behavior.
  • ● No admin UI for "re-register DCR client" — if a provider revokes our client_id, the existing REQUIRES_REAUTHORIZATION flow handles it. Manual SQL clear is the v1 escape hatch.
  • ● No DCR client rotation policy — RFC 7591 leaves expiry optional; we treat the client_id as long-lived and re-register on demand if needed.
